# 리뷰어 피드백 가이드

이 문서는 `1단계 - 사용자가 있는 Gemini 웹앱 출시하기` 리뷰를 쉽게 하기 위한 가이드입니다.

## 리뷰 의도

- 이번 미션의 핵심은 `AI와 함께 빠르게 만들기 -> 바로 배포하기 -> 사용자 반응으로 개선하기`를 반복하며
  **프로덕트 빌더로 일하는 방식**을 경험하는 것입니다.
- 완벽한 코드/설계보다, `나에게 실제로 필요한 문제`를 정의하고 동작하는 앱을 세상에 내놓는 경험을 더 중요하게 봐주세요.
- Gemini Canvas를 활용한 프롬프트 반복 개선, 그리고 페어 프롬프트 릴레이 같은 AI 시대 협업 방식까지 함께 확인해 주세요.
- 모든 것을 지적하기보다, 제출자가 다음 실험을 더 빠르게 할 수 있게 만드는 피드백을 남기는 것이 목표입니다.

## 리뷰 순서

1. 제출된 4개 앱의 배포 링크를 열어 기본 동작을 확인합니다.
2. 카테고리(유틸리티/게임/학습/페어 프롬프트 릴레이)가 모두 충족됐는지 확인합니다.
3. 최소 1개 앱의 AI 기능이 실제로 동작하고 사용자 가치가 있는지 확인합니다.
4. AI 사용 일지에서 초기 프롬프트, 개선 반복, 효과적 전략이 구체적인지 확인합니다.
5. 크루가 요청한 "피드백 받고 싶은 포인트"를 반영해 코멘트를 작성합니다.
6. 핵심 기능 동작이 확인되면 `Approve` 후 `merge`합니다.

## 우선순위와 리뷰 액션

- 이 미션에서는 `Request Changes` 대신 코멘트 기반 리뷰를 사용합니다.
- `필수`: 요구사항 미충족/동작 오류. 재현 경로와 기대 동작을 코멘트로 남기고, 반영 전까지 `merge`를 보류합니다.
- `권장`: 품질/명확성 개선. 코멘트로 제안하고, 다음 단계에서 실험 가능한 방향을 제시해주세요.
- `제안`: 선택 가능한 아이디어. 부담을 주지 않는 확장 아이디어로 남겨주세요.
- 핵심 기능이 동작하면 코멘트 작성 후 `Approve`하고 `merge`합니다.

## 검증 매트릭스

| 검증 항목 | 우선순위 | 확인 위치 | 리뷰어 액션 |
| --- | --- | --- | --- |
| 배포 링크가 열리지 않거나 핵심 기능이 동작하지 않음 | 필수 | 앱별 배포 링크 | 수정 요청 코멘트 + merge 보류 |
| 4개 카테고리(유틸리티/게임/학습/페어 릴레이) 중 누락이 있음 | 필수 | 제출 양식의 카테고리 목록 | 수정 요청 코멘트 + merge 보류 |
| 최소 1개 앱에 AI 기능이 없거나, 있어도 동작하지 않음 | 필수 | 앱 설명의 AI 기능 항목, 실제 동작 | 수정 요청 코멘트 + merge 보류 |
| AI 기능은 동작하지만 사용자 문제 해결 가치가 약함 | 권장 | 앱 사용 흐름, AI 결과물 | 개선 코멘트 후 merge |
| AI 사용 일지에 프롬프트 개선 과정이 추상적임 | 권장 | AI 사용 일지 1~3번 항목 | 개선 코멘트 후 merge |
| 제출자가 요청한 피드백 포인트를 다루지 않음 | 권장 | AI 사용 일지 5번 항목 | 보완 코멘트 후 merge |
| UI 디테일/확장 아이디어 | 제안 | 전반 | 선택형 제안 후 merge |

## 코멘트 작성 원칙

- 한 PR당 `핵심 2~3개 코멘트`면 충분합니다.
- 단, `필수` 이슈는 개수 제한 없이 우선 전달해 주세요.
- `좋았던 점 1개 + 개선 제안 1~2개 + 확인 질문 1개` 구조를 권장합니다.
- 추상 표현보다 확인 가능한 기준으로 작성해 주세요.
- 예: "사용성이 아쉽다" 대신 "모바일에서 주요 버튼이 화면 아래로 밀려 클릭이 어렵다"

## 코멘트 예시

```md
- 좋았던 점: 실제 사용 맥락(통학 중 빠르게 쓰는 메모 앱)이 명확해서 기능 선택 이유가 잘 보였습니다.
- 필수 확인: 배포 링크 접속 시 메인 화면이 비어 있습니다. (재현: Safari iOS 17, 첫 접속) 최소한 빈 상태 안내 문구가 보이도록 확인 부탁드립니다.
- 개선 제안: 
  - 완료/미완료 필터를 추가하면 일상 사용성이 더 좋아질 것 같습니다.
  - 요약 결과를 3줄/5줄 두 가지 모드로 제공해 사용자 선호를 비교해보면 좋겠습니다.
- 질문: AI 요약 결과가 길어질 때도 읽기 쉽도록 길이 제한이나 포맷 규칙을 둘 계획이 있을까요?

```

## 리뷰 마무리 체크

- [ ] `필수/권장/제안` 우선순위에 맞는 액션으로 코멘트를 남겼다.
- [ ] `Request Changes` 없이 코멘트 기반으로 리뷰를 진행했다.
- [ ] 제출자가 요청한 피드백 포인트를 반영했다.
- [ ] 핵심 기능 동작 확인 후 `Approve` 및 `merge`를 진행했다.
- [ ] 다음 실험으로 이어지는 개선 제안을 최소 1개 남겼다.
